# 🚀 现代 Git 提交与版本管理规范指南

为了保持代码提交历史（Commit History）的清晰可读，告别乱麻般的“修改xxx”，我们需要将**日常开发提交（Commit）**与**阶段性版本（Tag）**解耦。本指南为你（九月）定制，可作为你的技术外脑长期查阅。

---

## 🔑 一、Commit 提交信息规范 (Angular 规范简化版)

每次提交代码时，不要再写模糊的“修改xx”。统一使用以下格式：
`🔑 <类型>: <简短描述>`

### 常用 5 大提交类型

| 类型 | 说明 | 适用场景示例 |
| :--- | :--- | :--- |
| **`feat`** | **新功能 (Feature)** | `feat: 增加用户微信登录功能`<br>`feat: 导出的 Excel 增加合计行` |
| **`fix`** | **修 Bug (Bug Fix)** | `fix: 修复深色模式下字体看不清的问题`<br>`fix: 修复点击保存按钮重复提交的 bug` |
| **`docs`** | **修改文档 (Documentation)** | `docs: 更新 README.md 中的本地部署教程`<br>`docs: 补充配置文件中的参数注释` |
| **`style`** | **样式调整与格式化 (Style)** | `style: 使用 Prettier 格式化代码`<br>`style: 调整主页按钮的左边距，不改变业务逻辑` |
| **`refactor`** | **代码重构 (Refactor)** | `refactor: 提取公共组件以减少冗余代码`<br>`refactor: 优化数据库查询语句，提升响应速度` |

---

## 📌 二、版本管理核心逻辑 (Commit 别写版本号)

1. **Commit 只负责“过程”**：记录每一次微小的代码细胞演进。
2. **Tag（标签）才负责“结果”**：版本号（如 `v1.0`、`v1.1`）应该作为标签挂在 Commit 旁边，而不是写进消息里。

### 💡 规范后的 Git 历史示意图

```text
 🟢 (Commit 5) feat: 新增数据导出功能
 🟢 (Commit 4) fix: 修复输入框错位问题       🏷️ [Tag: v1.1]  <-- 1.1 版本到此为止
 🟢 (Commit 3) docs: 补充了部署文档
 🟢 (Commit 2) refactor: 优化了查询逻辑
 🟢 (Commit 1) feat: 初始化项目架构          🏷️ [Tag: v1.0]  <-- 1.0 版本里程碑
```

## 🛠️ 三、标准开发与发布流水线（日常照着做）

### 1. 日常开发：疯狂提交各类 Commit
```bash
git add .
git commit -m "feat: 增加购物车删除商品功能"
```

### 2. 发布新版本：打上 Tag 戳
当累积的小更新足够多，决定发布 `v1.1` 时：
```bash
# 1. 在当前最新提交上打一个版本标签
git tag -a v1.1 -m "发布 1.1 版本，优化体验并修复已知问题"

# 2. 将标签推送到远程仓库（如 GitHub/Gitee）
git push origin v1.1
```

### 3. 跨版本查账：一眼看出版本间更新了什么
想知道 `v1.0` 到 `v1.1` 之间到底更新了哪些“小修改”？
```bash
git log v1.0..v1.1 --oneline
```
**终端输出效果：**
> `a1b2c3d fix: 修复输入框错位问题`
> `e5f6g7h docs: 补充了部署文档`
> `j9k8l7m refactor: 优化了查询逻辑`

---

## 📝 总结你的新习惯
> **写代码时**：只想这次是 `feat` 还是 `fix`。
> **发布版时**：随手敲一句 `git tag v1.X`。
> **查账看版本区别时**：用 `git log v1.A..v1.B`。
